이 문서는 감사보고서를 실제 보완 작업으로 전환하기 위한 실행 계획이다. 목표는 "문제 목록"을 늘리는 것이 아니라, 유지보수와 신규 개발이 쉬운 베이스를 순차적으로 만드는 것이다. 따라서 모든 단계는 개발자 작업, PM 의사결정, 산출물, 완료 기준을 함께 가진다.

실행 원칙

Sellerhub는 전면 재작성보다, 폭발 반경이 큰 리스크를 먼저 줄이고 그 위에 개발 기반을 정리하는 방식이 현실적이다.

왜 이 순서인가

  • 보안과 외부 연동 리스크를 먼저 줄이지 않으면, 이후 정리 작업도 안전하지 않다.
  • 테스트가 믿을 수 없으면 리팩토링 속도를 낼 수 없다.
  • 부트스트랩과 엔트리포인트가 정리되지 않으면 신규 기능도 계속 레거시 패턴으로 추가된다.
  • 인코딩과 문법 현대화는 마지막이 아니라, 기반이 어느 정도 정리된 뒤에 채널 단위로 수행해야 실패 비용이 낮다.

이번 문서의 목표

  • 개발자: 어디부터 손대야 하는지, 어떤 산출물을 남겨야 하는지 명확히 한다.
  • PM: 어떤 단계가 선행 조건인지, 어디서 일정과 리스크를 관리해야 하는지 알 수 있게 한다.
  • 조직: 유지보수 가능한 공통 기준을 문서와 코드 구조로 남긴다.
전면 재작성 금지 채널 단위 분리 테스트 기반 확보 운영 영향 최소화

권장 실행 순서

단계는 반드시 순차적으로 진행하는 것을 권장한다. 특히 1~3단계는 선후 관계가 강하다.

단계 핵심 목적 PM 초점 개발 초점
1. 운영 리스크 차단 시크릿, 웹훅, 위험 호출을 통제 가능한 상태로 만든다 영향 범위와 교체 정책 확정 하드코딩 제거, 설정 분리
2. 테스트 기반 재정비 안전하게 수정 가능한 최소 회귀 안전망을 만든다 우선 보호할 핵심 플로우 선정 테스트 부트스트랩 분리, 외부 IO 차단
3. 부트스트랩 분해 공통 초기화의 폭발 반경을 줄인다 채널별 영향도 승인 세션/DB/웹 초기화 분리
4. 엔트리포인트/화면 계층 정리 개발자가 기능을 추가하기 쉬운 구조를 만든다 우선 채널 결정 데이터 준비 로직 이동
5. 현대화 기반 구축 UTF-8, short tag 제거, 규칙 문서화로 장기 유지보수 기반을 만든다 변환 범위와 일정 조율 채널 단위 마이그레이션

단계별 실행 계획

각 단계는 바로 티켓으로 쪼갤 수 있도록 목적, 작업, 산출물, 완료 기준으로 정리했다.

1단계. 운영 리스크 차단

보안과 외부 연동 리스크를 먼저 통제한다. 이 단계가 끝나야 이후 정리 작업을 안심하고 진행할 수 있다.

Phase 1
최우선 예상 성격: 단기 집중 작업 선행 이유: 즉시 사고 가능성 차단

PM 해야 할 일

  • 운영 알림, 배치, 테스트 중 실제 외부 시스템과 연결된 목록을 확정한다.
  • 키 회전 필요 여부와 운영 영향 시간을 결정한다.
  • 민감정보 관리 방식을 환경변수 또는 별도 설정 파일 중 하나로 고정한다.

개발자 해야 할 일

  • 코드/테스트에 박힌 웹훅 URL, 토큰, 시크릿을 수집하고 제거한다.
  • 외부 연동 설정 로더를 한 곳으로 모은다.
  • 테스트에서 실제 외부 호출을 막고 mock/stub 경로를 만든다.

산출물 / 완료 기준

  • 민감정보 제거 PR
  • 설정 키 목록 문서
  • 완료 기준: 저장소 grep 기준 하드코딩된 운영 웹훅/시크릿이 남지 않는다

2단계. 테스트 기반 재정비

이 단계의 목적은 커버리지를 높이는 것이 아니라, 구조 정리 중 깨지면 안 되는 핵심 흐름을 보호하는 것이다.

Phase 2
필수 기반 대상: 주문/로그인/정산/핵심 운영 플로우

PM 해야 할 일

  • 절대 깨지면 안 되는 3~5개 핵심 플로우를 선정한다.
  • 진단용 테스트와 배포 차단용 테스트를 구분해 승인한다.

개발자 해야 할 일

  • tests/bootstrap.php에서 서비스 전체 부트스트랩 의존을 줄인다.
  • 실DB 직접 변경 테스트와 읽기 전용 테스트를 분리한다.
  • 외부 IO, 웹훅, 배치 호출은 모두 test double로 치환한다.

산출물 / 완료 기준

  • 테스트 전략 문서
  • 테스트 전용 부트스트랩
  • 완료 기준: 핵심 플로우에 대해 로컬/CI에서 반복 실행 가능한 테스트 세트가 존재한다

3단계. 공통 부트스트랩 분해

가장 중요한 구조 작업이다. 신규 개발과 유지보수가 쉬운 베이스를 만들려면 이 단계를 피해갈 수 없다.

Phase 3
핵심 리팩토링 대상: shop/lib/library.php

PM 해야 할 일

  • 채널별 영향도 우선순위를 정한다. 예: admin → whub → m2.
  • 한 번에 전체 교체하지 않고 단계 배포 원칙을 확정한다.

개발자 해야 할 일

  • 공통 부트스트랩을 최소 3개 층으로 분리한다. 예: 환경/인프라, 세션/인증, 웹 응답 초기화.
  • CLI, 배치, 웹에서 공통으로 쓰는 초기화와 웹 전용 초기화를 분리한다.
  • 엔트리포인트가 필요한 초기화만 호출하도록 의존 방향을 바꾼다.

산출물 / 완료 기준

  • 부트스트랩 구조 문서
  • 분리된 초기화 모듈
  • 완료 기준: 공통 부트스트랩 하나를 수정하지 않고도 개별 채널 초기화가 변경 가능하다

4단계. 엔트리포인트와 화면 계층 정리

개발자가 기능을 추가할 때 가장 자주 만나는 층을 단순하게 만든다. 이 단계부터 체감 유지보수성이 올라간다.

Phase 4
생산성 개선 대상: whub/header.php, m2/_header.php 계열

PM 해야 할 일

  • 가장 변경이 잦은 채널과 기능을 우선 대상에 둔다.
  • 개선 중 신규 기능 작업과 충돌하지 않도록 범위를 분할한다.

개발자 해야 할 일

  • 헤더/화면 파일의 DB 접근, 세션 분기, 캐시 접근을 서비스 또는 presenter 계층으로 이동한다.
  • 채널별 공통 데이터 준비 코드를 재사용 가능한 모듈로 분리한다.
  • 파일 위치만 봐도 책임이 드러나는 구조로 정리한다.

산출물 / 완료 기준

  • 채널별 엔트리 구조 문서
  • 대표 화면 리팩토링 PR
  • 완료 기준: 대표 화면 파일에서 직접 DB/도메인 조회가 유의미하게 감소한다

5단계. 현대화 기반 구축

이 단계는 마지막 마감 작업이 아니라, 이후 개발 비용을 계속 낮추기 위한 장기 기반이다.

Phase 5
장기 기반 대상: UTF-8, short tag 제거, 코딩 규칙 표준화

PM 해야 할 일

  • 채널별 전환 순서와 검수 범위를 정한다.
  • 한 번에 전체를 바꾸지 않고 배포 가능한 단위로 관리한다.

개발자 해야 할 일

  • 신규 코드 UTF-8, <?php 강제 규칙을 먼저 도입한다.
  • 변경이 많은 채널부터 short tag 제거를 시작한다.
  • 문서화된 구조 규칙과 PR 체크리스트를 만든다.

산출물 / 완료 기준

  • 개발 규칙 문서
  • 채널 단위 변환 작업표
  • 완료 기준: 신규/수정 코드가 더 이상 레거시 패턴을 추가하지 않는다

PM / 개발자 역할 분담 기준

이 프로젝트는 기술 문제처럼 보이지만, 실제로는 의사결정과 범위 관리가 품질을 좌우한다.

PM이 반드시 잡아야 할 것

  • 이번 정리에서 보호할 핵심 비즈니스 플로우 우선순위
  • 운영 키 회전, 외부 연동 영향, 배포 가능 시간대
  • 채널별 개선 순서와 동시에 열 수 있는 작업 개수
  • 리팩토링 중 신규 기능 요구를 어느 수준까지 허용할지

개발자가 반드시 남겨야 할 것

  • 각 단계별 구조 변경 이유와 영향 범위 문서
  • 새로운 공통 모듈의 사용 규칙
  • 테스트 가능한 최소 사례와 재현 절차
  • 다음 단계 작업자가 이어받을 수 있는 체크리스트

후속 백로그 운영 방식

문서를 만든 뒤 가장 흔한 실패는 "좋은 계획이지만 실제 티켓으로 연결되지 않는 것"이다. 아래 방식으로 운영하는 것을 권장한다.

백로그 종류 포함 항목 비고
즉시 실행 티켓 시크릿 제거, 테스트 외부 호출 차단, 위험 exec 정리 1단계 안에서 종료 가능한 작업만 분리
기반 작업 티켓 테스트 부트스트랩, 공통 설정 로더, 초기화 분리 2~3단계 핵심
채널 정리 티켓 whub, m2, admin 계열 엔트리포인트 정리 한 채널씩 닫는 방식 권장
장기 현대화 티켓 UTF-8 전환, short tag 제거, 규칙 표준화 기반 작업과 병행하지 말고 단계 종료 후 진행

실행 후 기대 상태

아래 상태에 가까워질수록 유지보수와 신규 개발이 쉬워진다.

개발자 관점

  • 파일을 열면 책임이 드러난다.
  • 기능 수정 전에 실행할 수 있는 테스트가 있다.
  • 신규 코드가 레거시 관성에 끌려가지 않는다.

PM 관점

  • 어떤 단계가 일정 리스크인지 설명 가능하다.
  • 운영 영향과 기술 부채 정리를 분리해 관리할 수 있다.
  • 기반 작업이 단순 리팩토링이 아니라 개발 속도 개선이라는 점이 명확하다.

조직 관점

  • 보안, 테스트, 구조에 대한 공통 규칙이 생긴다.
  • 온보딩 비용이 줄고, 특정 개발자 의존도가 내려간다.
  • 레거시 추가 속도보다 정리 속도가 빨라지는 전환점이 생긴다.
